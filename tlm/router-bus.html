Designing an Advanced SystemC TLM-2.0 Bus Router: Handling Dynamic Address Mapping and Device-Specific Latencies

In electronic system-level (ESL) modeling, designing an interconnect bus router presents a classic engineering trade-off: flexibility versus structural awareness. A production-grade simulator must map incoming multi-core requests dynamically while handling unique hardware traits—such as the slow write latency of Flash memory versus the instant execution velocity of Data SRAM.

Hardcoding address map segments inside the decode cycle creates unmaintainable design bottlenecks. Below, we step through a comprehensive, scalable design pattern that implements an adaptable vector tracking architecture capable of adjusting transaction pathways and simulating distinct hardware processing loops on the fly.

The Design Breakthrough: Instead of executing a static memory decoding block, we upgrade our router to utilize an object-oriented vector structure. This allows us to assign a Device ID status wrapper to distinct target segments. The router can then apply dynamic latency offsets while remaining a completely reusable class component.

The Structural Blueprint: Why Basic Decoding Fails

An intermediate verification layout often handles target processing by evaluating boundaries using a primitive conditional index pattern, like this:

int decode_address(uint64_t addr) {
    if (addr < 0x40000) return 0;  // 0x00000 - 0x3FFFF -> Flash Memory Device
    return 1;                     // 0x40000+          -> Data SRAM
}

While this configuration successfully flags if a target coordinate points to Flash (Device 0) or SRAM (Device 1), it leaves the system brittle. If an engineering update inserts a third element—like a Boot ROM or a hardware peripheral—the foundational logic breaks, requiring full codebase refactoring and recompilation.

The Advanced Solution: The Parametric Vector Matrix

To preserve reusability without sacrificing the capability to implement specific processing behaviors, we register target attributes inside a structured look-up block vector. Here is the refactored, production-ready SystemC implementation:

#include <systemc.h>
#include <tlm.h>
#include <tlm_utils/multi_passthrough_target_socket.h>
#include <tlm_utils/multi_passthrough_initiator_socket.h>
#include <vector>

struct AddrMapping {
    uint64_t start_address;
    uint64_t end_address;
    int target_socket_index; // Acts as the explicit Device ID identifier
    std::string device_name;
};

class ParametricRouter : public sc_module {
public:
    tlm_utils::multi_passthrough_target_socket<ParametricRouter>    cpu_inputs;
    tlm_utils::multi_passthrough_initiator_socket<ParametricRouter> mem_outputs;

    SC_HAS_PROCESS(ParametricRouter);
    ParametricRouter(sc_module_name name) 
        : sc_module(name), cpu_inputs("cpu_inputs"), mem_outputs("mem_outputs") {
        
        cpu_inputs.register_b_transport(this, &ParametricRouter::b_transport);
    }

    void register_device(uint64_t start, uint64_t end, int device_id, std::string name) {
        AddrMapping mapping = {start, end, device_id, name};
        m_routing_table.push_back(mapping);
    }

    virtual void b_transport(int id, tlm::tlm_generic_payload& trans, sc_time& delay) {
        uint64_t global_addr = trans.get_address();
        int matched_idx = decode_address(global_addr);
        
        if (matched_idx == -1) {
            trans.set_response_status(tlm::TLM_ADDRESS_ERROR_RESPONSE);
            return;
        }

        int device_id = m_routing_table[matched_idx].target_socket_index;
        uint64_t local_offset = global_addr - m_routing_table[matched_idx].start_address;
        trans.set_address(local_offset);

        // Execute Device-Specific Latency Processing Logic inside the central hub
        if (device_id == 0) {
            // Specialized handling for FLASH MEMORY: Inject access latency penality
            delay += sc_time(50, SC_NS); 
            cout << "@ " << sc_time_stamp() << " [Router] Slow Flash Wrapper Mode Triggered\n";
        } else if (device_id == 1) {
            // Specialized handling for DATA SRAM: High speed operational routing
            delay += sc_time(1, SC_NS);
        }

        // Forward transaction payload to the validated destination target socket interface
        mem_outputs[device_id]->b_transport(trans, delay);
                trans.set_address(global_addr); // Crucial: Re-align mapping state before return
    }

    virtual bool get_direct_mem_ptr(int id, tlm::tlm_generic_payload& trans, tlm::tlm_dmi& dmi_data) {
        uint64_t global_addr = trans.get_address();
        int matched_idx = decode_address(global_addr);
        
        if (matched_idx == -1) return false;

        int device_id = m_routing_table[matched_idx].target_socket_index;
        uint64_t base_addr = m_routing_table[matched_idx].start_address;
        uint64_t local_offset = global_addr - base_addr;

        trans.set_address(local_offset);
        bool success = mem_outputs[device_id]->get_direct_mem_ptr(trans, dmi_data);
        trans.set_address(global_addr);
        
        if (success) {
            // Re-align DMI slice boundaries back up to global system mapping coordinate spaces
            dmi_data.set_start_address(dmi_data.get_start_address() + base_addr);
            dmi_data.set_end_address(dmi_data.get_end_address() + base_addr);
        }
        return success;
    }

private:
    std::vector m_routing_table;

    int decode_address(uint64_t addr) {
        for (size_t i = 0; i < m_routing_table.size(); ++i) {
            if (addr >= m_routing_table[i].start_address && addr <= m_routing_table[i].end_address) {
                return i;
            }
        }
        return -1; // Address Mapping Range Fault
    }
};

How the Dynamic Lookup Array Operates

By leveraging this structural pattern, the memory maps are initialized outside the tracking architecture during system configuration. Let's trace how the internal system maps these elements:

Vector Array Index Target Memory Boundary Assigned Device ID Router Internal Behavior Logic
Index 0 0x00000 to 0x3FFFF Device 0 (Flash) Appends a 50ns wait-state synchronization penalty
Index 1 0x40000 to 0x7FFFF Device 1 (SRAM) Appends a baseline 1ns structural access delay
Index 2 0x80000 to 0x9FFFF Device 2 (Boot ROM) Applies read-only execution constraints (Custom Extensible)
